# RegimeRadar — Refined Action Plan
_Companion to `regime-radar-audit-2026-05-20.md`. This is a strengthening/refinement pass focused on actionability, not a fresh audit. Date: 2026-05-20._

> Note on judgment: where the requested reframes were too soft, I kept the substance and only adjusted the phrasing. Specific pushbacks are flagged inline as **[Pushback]**.

---

## 1. Revised executive summary

RegimeRadar is already a strong portfolio/research artifact — and, unlike most portfolio projects, it has a real **product shell**: live deployment, a coherent multi-page structure, genuine model outputs, and a legitimately strong Scenario Explorer. The *plumbing* of a recurring-use tool is largely built.

What's missing is narrow but load-bearing: a **trustworthy headline**, a **reason to return**, and an **interpretation layer** that lets a non-technical user understand the product in ~10 seconds. The remaining distance is small in engineering effort (≈1–2 weeks of focused work) but it is exactly the part that separates "impressive demo" from "tool people check."

Highest leverage, in order: (1) **trust** — make the headline evidenced and coherent, restore live drivers; (2) **recurring use** — daily "what changed" + an alert/feed loop; (3) **interpretation** — first-use clarity. Net-new *showcase* features are not the lever right now, but several genuine features (empirical reliability band, daily shareable card, compare-to-past, regime calendar) double as trust/retention/distribution and are worth building.

## 2. Revised blunt final assessment

- **What it is:** a top-decile portfolio project that has already crossed the line most never reach — deployed, structured, built on real models, with one genuinely good interactive (Scenario Explorer).
- **How close to recurring use:** closer than most portfolio projects in *infrastructure*, but it has **zero retention surface** today and the live headline isn't trustworthy. So it's roughly one to two weeks of the *right* work from a minimum viable recurring tool — not months — but that work is specifically **trust + return loop**, not more UI.
- **Biggest gap:** trust first, return-reason second, interpretation third.
- **[Pushback] on "not close":** softened in phrasing, not in substance. "Has a product shell" is real and credited. But don't let it blur into "almost a real tool" — the missing 20% (trust + retention) is the whole reason a tool gets used. **Close in effort, far in function.** Hold both at once.

## 3. Tier 1–4 action stack

### Tier 1 — Must fix for trust (credibility blockers; do first)
- **T1.1 Headline coherence.** Reconcile `transition_risk` with the regime probabilities. Never show "81% chance conditions worsen" next to "0.02% turbulent / VIX 18 / trend up / stable" without reconciling them in the copy.
- **T1.2 Empirical reliability inline** _(deep fix + a real feature)._ From OOF data, precompute "P(actually worsened | model read ≈ p)" and show it with the number: "historically, when the model read ~80%, conditions worsened X% of the time (n=…)." Converts an abstract probability into an evidenced one and defuses the 0.81 problem directly.
- **T1.3 Out-of-range flagging.** If a live reading sits beyond the model's evaluated OOF range (the published sweep tops out at 0.30), say so plainly rather than stating 0.81 flat.
- **T1.4 Restore live `top_drivers`.** `/current-state` returns `[]` in production; compute SHAP live or reuse `/model-drivers` `local_explanation`. Stop the silent fallback-to-global presented as "what's pushing risk now."
- **T1.5 Fix Event Replay's threshold-flattering.** Metrics computed at a 10% threshold make every event look "caught" (37 alert days for 2008). **[Pushback]** Recategorizing Event Replay as trust-building (per §below) *raises* its honesty bar — fix this.
- **T1.6 README↔UI drift.** Finnhub price-card (doesn't exist in frontend), "5 days" vs "this week", route `/model-drivers` vs title "Signal Breakdown", "66 smoke tests" (= whole pytest suite). Small, but it's literal misrepresentation.
- **Cross-cutting reliability (fold in here):** CI running the existing ~149 tests on push; monthly retrain + drift/eval gate (the frozen-2026-04-24 model is the root cause behind T1.1); precompute `/historical-state` & `/model-drivers`; lock down open `POST /refresh-data` + scope CORS. Not user-facing, but they *protect* trust.

### Tier 2 — Must add for recurring usefulness (the return loop)
- **T2.1 True "what changed since yesterday."** Replace the 5-minute delta with a real daily diff from the committed nightly snapshots (regime change, risk move, top mover).
- **T2.2 RSS/Atom feed** of regime changes + threshold crossings — free, no DB, generated by the existing nightly GitHub Action.
- **T2.3 Email alert** on regime change (free tier; opt-in via a simple form → nightly job sends).
- **T2.4 Daily shareable state card** _(feature + distribution; see §6)._

### Tier 3 — Must improve for interpretation / adoption
- **T3.1 10-second first-use clarity** (see §4).
- **T3.2 Templated insights** → clearly-labeled heuristic scaffolding; upgrade to model-derived where cheap. **[Pushback]** Exception: the "what would raise risk further" block implies forward-looking model reasoning — make it genuinely model-derived (perturb each feature, read direction) or reframe as generic education. Labeling alone isn't enough there. The driver-interpretation map is fine as labeled heuristic.
- **T3.3 History upgrades:** auto-annotate regime transitions, compare-to-past-date, regime calendar heatmap. (History is underpowered, **not** low-value — these unlock its latent value.)
- **T3.4 Mobile responsive layout** (collapsible sidebar, responsive grids).

### Tier 4 — Nice later additions (only with a returning user)
- Multi-asset (bonds/gold/sectors); accounts/watchlists with custom thresholds; intraday; backtested strategy overlay; product analytics/instrumentation. Real features — just don't build on spec.

## 4. Onboarding / first-use clarity (its own product problem)

**What a first-timer understands in 10 seconds today:** there's a market "state" (Elevated), a big % number, it's "live," and it looks serious and clean. The color coding reads.

**What they still don't understand in 10 seconds:** what "transition risk" means; why 81% sits next to a calm VIX and an up trend; whether 81% is high, normal, or low; what they're supposed to *do*; how much to trust it; and what changed since last time. The glossary is buried in the sidebar footer ("HOW TO READ THIS").

**Where interpretation effort is still too high:** the user has to cross-reference four numbers to even notice they conflict; "this week" (UI) vs "5 trading days" (docs); calibration — the project's whole credibility pitch — is never mentioned in the UI.

**What would need to change so a non-technical user immediately gets it:**
- One plain sentence at the very top: _"The US stock market is in an **Elevated** state. There's an **[X]** chance conditions get more stressed in the next week — **[higher than / about / lower than] usual**."_
- **Anchor the number, never show it naked** — always against base rate / percentile / track record.
- A one-glance **trust cue**: "the model's track record at this level: worsened X% of the time."
- A persistent **"what changed since yesterday"** line.
- Promote "How to read this" from the footer into a first-visit inline hint.

## 5. Decision usefulness (JTBD — what behavior actually changes)

- **JTBD it supports today (weakly):** "Help me decide how much attention to pay to the market this week." Even this is undercut by the incoherent headline.
- **JTBD it fails to support:** "Tell me when something changes so I don't have to check" (no alerts); "help me size/adjust risk" (no actionable, backtested threshold); "give me cross-asset context" (single asset).
- **What concrete behavior changes because it exists, honestly:** today, almost none — there's no trigger to act and no push to bring the user back.
- **What would make it change behavior:** (a) a push — "regime changed to Elevated today" → user opens/acts; (b) an evidenced threshold — "at this level, markets worsened X% within a week historically" → user calibrates attention/risk; (c) a daily one-glance state so checking becomes a 5-second habit. **The retention loop and the decision loop are the same loop** — build it once.

## 6. Distribution / reach (how utility reaches the user — not growth hacks)

**[Pushback] on sequencing:** distribution only creates value *after* Tier 1 (trust) and once there's a real daily artifact. Pushing an incoherent 81% headline to a feed accelerates distrust — negative value. Framed right, distribution is **the same build as the return loop, pointed outward.**

Realistic free channels, all fed by **one** daily artifact:
- **RSS/Atom feed** of regime changes (zero infra; power users + aggregators subscribe).
- **Daily email summary** (free tier).
- **Shareable daily card** — auto-generated PNG/OG image: _"RegimeRadar — May 20: ELEVATED · risk 81% (↑ from Calm)."_ Works as a social preview, a manual share, and the thing the email/feed embeds.
- **Public regime-change changelog** — a dated list page that doubles as the RSS source.
- **Good OG/Twitter meta** so any shared link renders the daily card.

One build (card + diff) → feed + email + changelog + social preview.

## 7. If I only did 3 things (specific, free, high-leverage)

1. **Make the headline evidenced, not just stated.** Precompute from OOF a "P(actually worsened | model read ≈ p)" lookup and show it inline with base-rate/percentile context; reconcile with the regime probabilities; flag out-of-range readings; and restore live `top_drivers`. Never display a bare 81%. _(Fixes the #1 trust killer. ~days.)_
2. **Ship the daily return loop as one artifact.** Build "what changed since yesterday" + a generated daily state card, and emit both an RSS feed and an optional email from the existing nightly GitHub Action. One build = retention + distribution. _(First real reason to return. ~days–1 week.)_
3. **Add CI + a monthly retrain/drift gate.** Run the existing ~149 tests on push; retrain monthly committing fresh artifacts; fail the build if calibration/Brier drifts past a bound. _(Fixes the frozen-model root cause behind #1 and makes the tests protect the product. ~1–2 days.)_

## Appendix — net-new features worth building (and which to resist)

**Build (each doubles as trust / retention / distribution):**
- Empirical reliability band (T1.2) — feature that fixes trust.
- Daily shareable state card (T2.4 / §6) — feature that fixes retention + reach.
- Compare-to-past-date + regime calendar heatmap (T3.3) — features that give History purpose.

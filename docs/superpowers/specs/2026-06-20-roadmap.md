## Guiding principle

Regime Radar is a **credibility product**: its value is one trustworthy number (current regime + transition risk). The original problem was that the number was **self-undermining** — live transition scores reach ~76% but the reliability table's validated range (`max_evaluated_p`) caps at ~0.30, so the UI disclaimed the score as a "directional stress signal, not a calibrated probability."

That problem is now **architecturally resolved** by the two-zone honesty feature (below): the number is split into a *validated* calibrated-probability zone (p ≤ `max_evaluated_p`, the only state that shows a "%") and an *unvalidated* stress/severity zone (`stress_in_support` / `stress_out_of_support`) that shows a severity tier and historical-rank language — never a calibrated-looking probability. The product no longer contradicts itself; it tells the truth about which zone it's in.

Polish and distribution amplify whatever trust the core number has. So the core number came first — and now stands on honest footing.

**Where things live / deploy:**
- Frontend → Vercel project `regime-radar`, the `frontend/` dir is the linked root. Pushing to `main` auto-deploys to production (canonical domain `regime-radar.vercel.app`). No manual `vercel` step needed.

- CI → `.github/workflows/ci.yml`: Backend tests (`pip install -r requirements.txt` then pytest) + Frontend build. **CI installs ONLY `requirements.txt`.**

**CI dependency convention (learned the hard way):** `requirements.txt` is the *only* thing CI installs. A top-level `import` of a package not in it will break pytest **collection** and abort the whole job, even though it passes locally. Keep heavy/optional deps (e.g. matplotlib, used only by `scripts/run_ceiling_diagnostic.py` for a diagnostic PNG) **lazy + optional** inside the function that needs them — do NOT add them to `requirements.txt` just to satisfy a research script.

**Intentional design decisions — do NOT "fix" these as bugs:**
- *Version-mismatch fallback shows a calibrated %:* in `src/api/risk_reading.py`, when the raw-score reference's `model_version` doesn't match the serving model, stress percentile/tier are suppressed and an in-support, above-ceiling reading falls back to `display_state="validated"` (so it DOES render a "%"). This is deliberate: with a stale rank reference we can't honestly rank severity, so we show the calibrated p with a loud log. Tested. The fix for a *real* mismatch is to re-run `build_raw_score_reference.py`, not to change this code.
- *`display_state` is derived support-first* (out_of_support → stress_in_support → validated). Don't reorder.
- *Current State is fallback-only* for `RiskReading` by design (Tier 3 lists the fuller treatment as future work).
